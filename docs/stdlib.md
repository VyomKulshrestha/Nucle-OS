# NucleScript Standard Library

`consensus_vote` and `protect` are NucleScript's two built-in functions --
generated by reading `nucle_lang/src/stdlib.rs`'s `builtin_functions()`
directly, so this page can't silently drift from what the compiler
actually resolves.

Unlike `simulate`/`synthesize`/`sequence` (dedicated grammar forms, kept
that way because their `confirm hardware` effect-confirmation semantics
are load-bearing enough to want a real grammar production), these two are
*just* functions that happen to ship with the compiler: each is a real
`FunctionTable` entry, resolved through the exact same lookup a call to
your own `fn` goes through -- arity checking, effect propagation, and
"did you mean X?" suggestions all come from that one shared path, not a
separate hardcoded case per built-in.

Both keep their existing keyword-sugar surface syntax (`consensus_vote(
source, coverage: N)`, `protect data for guarantee`) -- the parser
desugars each straight to an ordinary function call at parse time (see
`parser::parse_primary_expr`), so a user function and a call to either of
these produce the identical `Expr::FunctionCall` shape. `consensus_vote`
additionally needs one narrow, explicit type-inference special case
(`typeck::TypeChecker::infer_consensus_vote`) because its result
genuinely depends on its *argument values* (the source binding's inferred
error rate and the requested coverage), not just a fixed declared
signature -- the same reason real compilers special-case a handful of
true intrinsics (`size_of::<T>()` and friends) by name rather than
pretending they fit an ordinary statically-typed function.

## `consensus_vote`

```nuclescript
consensus_vote(source, coverage: N)
```

| | |
|---|---|
| **Effect** | `Pure` (always confirmed, no `confirm hardware`/`confirm physical_key` needed) |
| **Parameter `source`** | A probabilistic pool binding (produced by `simulate`/`synthesize`/`sequence`), in any profile/state |
| **Parameter `coverage`** | A multiplier literal (`10x`) -- how many independent coverage copies to vote across |
| **Result** | `Pool<Recovered, E%>`, where `E` is computed from `source`'s inferred error rate and `coverage` via `probabilistic::consensus_error_rate_percent` |

Majority-vote across `coverage` independently-noisy copies of the same
strand, correcting substitution/indel errors that Reed-Solomon parity
alone can't (parity reconstructs a strand that's entirely *missing*, not
one that came back *wrong but present*). See
[`nucle_ecc::consensus`](../nucle_ecc/src/consensus.rs) for the real
partial-order-alignment implementation this models the error-rate
reduction of.

```nuclescript
pool archive: DnaPool { codec: Ternary, redundancy: 3x, profile: Illumina }

let noisy: Pool<Illumina, 0.35%> = simulate archive under Illumina
let recovered: Pool<Recovered> = consensus_vote(noisy, coverage: 10x)
```

**Diagnostics specific to this call:**

| Code | Level | Meaning |
|------|-------|---------|
| `E-CONSENSUS-INVALID-SOURCE` | error | `source` isn't a probabilistic pool binding. |
| `E-CONSENSUS-NOOP-COVERAGE` | warning | `coverage: 1x` -- voting across a single copy doesn't reduce the error budget at all. |

(`E-FUNCTION-ARITY`/`E-FUNCTION-UNDECLARED` apply too, exactly as they
would for a miscalled user function -- see [`docs/errors.md`](errors.md).)

## `protect`

```nuclescript
protect data for guarantee
```

| | |
|---|---|
| **Effect** | `Pure` (always confirmed) |
| **Parameter `data`** | The value to protect (a `File`/binding name) |
| **Parameter `guarantee`** | The recovery guarantee to protect it under (typically a `Recovery`-typed parameter) |
| **Result** | Untyped (`None`) -- `protect` doesn't itself infer a `Pool<...>` type; it's typically the last expression before a `store` |

Marks a value as protected under a named recovery guarantee, most useful
inside a reusable function that takes the guarantee as a parameter rather
than hardcoding it:

```nuclescript
fn archive(data: File, target: Pool<Illumina>, guarantee: Recovery) returns DnaFile {
    let plan: DnaFile = protect data for guarantee
    store plan into target
}
```

(See [`docs/examples/archive_fn.nsl`](examples/archive_fn.nsl) for this
exact example, exercised end to end.)

## Why not a general-purpose stdlib module system yet

Both functions are compiled directly into `nucle_lang` rather than
distributed as a package (compare
[`docs/packages.md`](packages.md)'s `@nuclescript/*` presets, which *are*
plain NucleScript source you can read). That's deliberate for this pair:
their return-type inference is genuinely compiler-level (value-dependent,
not expressible as a plain declared signature), so they need to live
where `typeck`/`middle` can special-case them by name. A future built-in
whose behavior *is* fully expressible with an ordinary signature wouldn't
need this -- it could be a real `FunctionDecl` with a real body, or even
ship as a package.
