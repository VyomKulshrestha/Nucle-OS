# NucleScript Diagnostic Codes

Every diagnostic `nucle check` (and the playground API) reports carries a
stable `code` alongside its `level` (`error`/`warning`), `message`, and
`span` (`file:line:column`). This page documents what each code means, a
minimal repro, and the fix — generated by reading the actual
`TypeReport::error`/`warning` call sites in
[`nucle_lang/src/typeck.rs`](../nucle_lang/src/typeck.rs) (plus the lex/parse
codes in [`lib.rs`](../nucle_lang/src/lib.rs)), so this list can't silently
drift from what the compiler actually emits — if you add a new check, add
its code here too.

Codes are grouped by which declaration/operation they check, matching
`typeck.rs`'s own `check_*` function layout.

## Lex/parse

| Code | Level | Meaning |
|------|-------|---------|
| `E-LEX-ERROR` | error | Tokenizing failed — e.g. an unterminated string literal. The message wraps the lexer's own error text. |
| `E-PARSE-ERROR` | error | The token stream doesn't match any known grammar production — e.g. a missing keyword (`into`, `from`), unbalanced braces, or a malformed declaration. The message wraps the parser's own error text. |

**Fix:** both point at the exact token where lexing/parsing gave up — check the source line the span underlines for a typo, missing keyword, or unclosed quote/brace.

## Pools (`pool` declarations)

| Code | Level | Meaning |
|------|-------|---------|
| `E-POOL-DUPLICATE` | error | Two `pool` declarations use the same name in one program. |
| `E-POOL-LOW-REDUNDANCY` | warning | A pool declares `redundancy: 1x`. Critical files should use at least `2x` so Reed-Solomon has parity to recover from. |
| `E-POOL-UNSUPPORTED-CODEC` | warning | A pool declares `codec: Fountain`. The VFS backend only executes `Ternary` and `YinYang` end-to-end today. |
| `E-POOL-UNDECLARED` | error | A `simulate`/`synthesize`/`sequence` expression references a pool name that isn't declared (via `pool` or a prior probabilistic `let` binding). Includes a "did you mean X?" suggestion when a same-scope name is within edit distance 2. |

**Fix:** rename the duplicate pool, raise redundancy, switch to a supported codec, or declare the referenced pool before using it.

## Imports (`import { ... } from "..."`)

| Code | Level | Meaning |
|------|-------|---------|
| `E-IMPORT-UNKNOWN-SOURCE` | error | The `from "..."` package source isn't in the registry (`packages/registry.json`). |
| `E-IMPORT-UNKNOWN-ITEM` | error | The package exists, but doesn't export the named item. Includes a "did you mean X?" suggestion against that package's actual exports. |

**Fix:** check `nucle package list`/`nucle package inspect <name>` for the real package name and its exports.

## Bindings (`let` declarations)

| Code | Level | Meaning |
|------|-------|---------|
| `E-BINDING-DUPLICATE` | error | Two bindings (or a binding and a pool) share the same name. |
| `E-SYNTHESIS-UNCONFIRMED` | error | The binding's expression has a `Synthesis`/`Sequencing`/`Destructive` effect (directly, or transitively through a function call) but is missing `confirm hardware`. |
| `E-BINDING-TYPE-MISMATCH` | error | The binding's `Pool<State>` annotation doesn't match the state the expression actually produces. |
| `E-BINDING-ERROR-RATE-MISMATCH` | error | The binding's annotated error-rate percentage differs from the inferred rate by more than 0.01%. |

**Fix:** add `confirm hardware`, correct the annotation to match what the expression actually infers, or fix the upstream expression.

## Expressions (inside any binding/argument)

| Code | Level | Meaning |
|------|-------|---------|
| `E-CONSENSUS-INVALID-SOURCE` | error | `consensus_vote(source, ...)`'s `source` isn't a probabilistic pool binding (i.e. wasn't produced by `simulate`/`synthesize`/`sequence`). |
| `E-CONSENSUS-NOOP-COVERAGE` | warning | `consensus_vote(..., coverage: 1x)` — voting across a single copy doesn't reduce the error budget at all. |
| `E-CONSENSUS-INVALID-COVERAGE` | error | `consensus_vote`'s second argument isn't a coverage number. Only reachable by hand-constructing an AST that bypasses the parser's `consensus_vote(source, coverage: N)` grammar — see [`docs/stdlib.md`](stdlib.md). |
| `E-FUNCTION-UNDECLARED` | error | A function call references a name with no matching `fn` declaration, or a typo'd built-in (`consensus_vote`/`protect` — see [`docs/stdlib.md`](stdlib.md)). Includes a "did you mean X?" suggestion against declared function names. |
| `E-FUNCTION-ARITY` | error | A function call passes the wrong number of arguments. |
| `E-ARG-TYPE-INVALID` | error | An argument bound to a `Pool<...>`-typed parameter doesn't infer to a pool type at all. |
| `E-ARG-TYPE-MISMATCH` | error | An argument's inferred pool state doesn't match the parameter's declared state. |
| `E-VARIABLE-UNDECLARED` | error | A bare variable reference doesn't resolve to any binding, pool, strand, or sequence in scope. Includes a "did you mean X?" suggestion. |

**Fix:** declare the missing function/variable, match the call's argument count/types to the declaration, or fix the typo the suggestion points at.

## Functions (`fn` declarations)

| Code | Level | Meaning |
|------|-------|---------|
| `E-FUNCTION-DUPLICATE` | error | Two `fn` declarations share the same name. |
| `E-PARAM-DUPLICATE` | error | A function declares the same parameter name twice. |
| `E-RETURN-TYPE-MISMATCH` | error | A function declared to return `Pool<State>` either produces a different state, doesn't end in a `let` binding that produces a pool at all, or its body doesn't end in a `let` binding. NucleScript has no explicit `return` — the last statement's binding is the implicit return value. |

**Fix:** rename the duplicate, rename the duplicate parameter, or make the function body's last binding produce the declared return type.

## Strands (`strand` declarations)

| Code | Level | Meaning |
|------|-------|---------|
| `E-STRAND-DUPLICATE` | error | Two `strand` declarations share the same name. |
| `E-STRAND-INVALID-DNA` | error | The strand literal contains characters outside the DNA alphabet, or fails basic parsing. |
| `E-STRAND-CONSTRAINT-VIOLATION` | error | The strand violates a biological constraint (GC content outside 40–60%, a homopolymer run over 3, a hairpin-prone palindrome over 6nt, or length outside 150–200nt). |

**Fix:** rename the duplicate, fix invalid characters, or redesign the sequence to satisfy synthesis/sequencing constraints.

## Sequences (`seq`/`let ... = seq"..."` declarations)

| Code | Level | Meaning |
|------|-------|---------|
| `E-SEQUENCE-DUPLICATE` | error | Two sequence declarations share the same name. |
| `E-SEQUENCE-INVALID-DNA` | error | The sequence literal contains characters outside the DNA alphabet (beyond the allowed `-`/`_`/whitespace separators), or is empty after normalization. |
| `E-SEQUENCE-CONSTRAINT-VIOLATION` | error | The sequence violates its (more permissive than `strand`) biological constraints — GC content, homopolymer, or palindrome length. |

**Fix:** same as strands, but sequence literals allow 1–500nt and an 8nt palindrome threshold.

## Store operations

| Code | Level | Meaning |
|------|-------|---------|
| `E-STORE-SYNTHESIS-WITHOUT-SIMULATE` | warning | A non-`simulate` `store` has a `Synthesis` effect — real hardware submission, not a dry run. |
| `E-STORE-POOL-UNDECLARED` | error | `store ... into <pool>` references an undeclared pool. Includes a "did you mean X?" suggestion. |
| `E-STORE-CRITICAL-LOW-REDUNDANCY` | warning | A store tagged `"critical"` uses only `1x` redundancy. |
| `E-STORE-UNSATISFIABLE-RECOVERY` | warning | `expect recovery > 99.5%` is requested for a Nanopore-profile pool at `1x` or less coverage — statistically unreachable at that coverage. |
| `E-STORE-IMPLICIT-COVERAGE` | warning | A `simulate store` doesn't set `coverage` explicitly, so it silently inherits the redundancy value as the assumed coverage. |

**Fix:** prefix with `simulate` for a dry run, declare the target pool, raise redundancy for critical data, raise coverage or lower the recovery target, or set `coverage` explicitly.

## Delete operations

| Code | Level | Meaning |
|------|-------|---------|
| `E-DELETE-POOL-UNDECLARED` | error | `delete ... from <pool>` references an undeclared pool. Includes a "did you mean X?" suggestion. |
| `E-DELETE-UNCONFIRMED` | error | A `delete` is missing `confirm physical_key` — deletion is `Destructive` and irreversible, so it always requires explicit confirmation. |

**Fix:** declare the target pool, or add `confirm physical_key`.

## Retrieve operations

| Code | Level | Meaning |
|------|-------|---------|
| `E-RETRIEVE-POOL-UNDECLARED` | error | `retrieve from <pool>` references an undeclared pool. Includes a "did you mean X?" suggestion. |
| `E-RETRIEVE-UNINDEXED-FIELD` | warning | A `where` predicate filters on a field the VFS search backend doesn't index (only `tag`, `date`, `size`, `name`, `type` are indexed). |

**Fix:** declare the target pool, or filter on an indexed field.

## Control flow (`if` / `for`) and `assert`

`if`'s condition and `assert`'s condition are evaluated by the exact same
compile-time machinery (`typeck::TypeChecker::eval_condition`), so both
share this one set of diagnostic codes — the codes were renamed from an
earlier `E-IF-CONDITION-*` scheme (Step 4) to the current `E-CONDITION-*`
form specifically so a message about `assert`'s condition doesn't
misleadingly say "if condition" (Step 7).

| Code | Level | Meaning |
|------|-------|---------|
| `E-CONDITION-NOT-BOOLEAN` | error | An `if`/`assert` condition isn't a comparison or a `&&`/`\|\|`/`!` combination of comparisons — e.g. a bare pool binding with no comparison operator. Both are resolved entirely at compile time (see [`grammar.md`](grammar.md#control-flow-if--for)), so the condition must reduce to a concrete `bool` during type checking. |
| `E-CONDITION-NOT-NUMERIC` | error | One side of a comparison inside an `if`/`assert` condition isn't a number literal or a probabilistic pool binding's name. |
| `E-CONDITION-UNDECLARED` | error | A comparison inside an `if`/`assert` condition references a pool binding name that isn't declared. Includes a "did you mean X?" suggestion. |
| `E-ASSERTION-FAILED` | error | An `assert`'s condition evaluated to `false`. The message is the custom one from `assert condition, "message"` if given, else the generic `"assertion failed"`. Valid anywhere a declaration is, not just inside `test { ... }` — `nucle check` reports an always-false assertion anywhere in a program as a real bug; `nucle test` additionally groups every one that falls within a given `test` block's span into that test's pass/fail result (see [`docs/grammar.md`](grammar.md#testing-test--assert)). |

**Fix:** rewrite the condition as a comparison (`binding > 0.5`) or boolean combination of comparisons, ensure both sides of every comparison are a number literal or a declared probabilistic pool binding, fix any typo'd binding name, or fix whatever the assertion actually found wrong.

## Pipelines

| Code | Level | Meaning |
|------|-------|---------|
| `E-PIPELINE-UNSUPPORTED-CODEC` | warning | A pipeline's `encode` step uses `Fountain`, which the VFS backend doesn't execute end-to-end. |
| `E-PIPELINE-LOW-REDUNDANCY` | warning | A pipeline's `protect` step uses `1x` redundancy. |
| `E-PIPELINE-POOL-UNDECLARED` | error | A pipeline's `store` step targets an undeclared pool. Includes a "did you mean X?" suggestion. |
| `E-PIPELINE-STORE-BEFORE-ENCODE` | error | A pipeline has a `store` step with no preceding `encode` step. |
| `E-PIPELINE-STORE-WITHOUT-PROTECT` | warning | A pipeline stores without an explicit `protect` step. |

**Fix:** switch codecs, raise redundancy, declare the target pool, or reorder/add the missing pipeline steps.
